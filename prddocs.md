You are a senior Product Manager and System Architect with experience in building production-grade software systems.

Your task is to create a complete, industry-standard Project Requirements Document (PRD).

I will provide:
- Problem statement
- Target users
- (Optional) tech stack / constraints
- (Optional) project scope

---

### STEP 1: PROBLEM UNDERSTANDING
- Clearly restate the problem in your own words
- Identify core pain points
- Define why this problem matters (business + technical perspective)

---

### STEP 2: PROJECT OVERVIEW
- Project name
- Objective (1–2 lines)
- Key goals (measurable outcomes)
- Non-goals (what this project will NOT do)

---

### STEP 3: USER ANALYSIS
- Target users (personas)
- User pain points
- User journeys (step-by-step interaction flow)

---

### STEP 4: FEATURE REQUIREMENTS (CORE + MVP)
Break into:
- Core Features (must-have)
- Secondary Features (good-to-have)

For each feature:
- Description
- Inputs (what data user provides)
- Processing (what system does)
- Outputs (what user gets)
- Edge cases

---

### STEP 5: FUNCTIONAL REQUIREMENTS
- System behaviors (API-level or logic-level clarity)
- Data flow (frontend → backend → database → response)
- Validation rules

---

### STEP 6: NON-FUNCTIONAL REQUIREMENTS
- Performance expectations
- Scalability considerations
- Security requirements
- Reliability and availability

---

### STEP 7: SYSTEM ARCHITECTURE (TEXT FORMAT)
- High-level architecture breakdown
- Components:
  - Frontend
  - Backend
  - Database
  - External services (if any)
- Data flow explanation

---

### STEP 8: TECH STACK JUSTIFICATION
- Suggested technologies
- Why each is chosen (not just naming tools)

---

### STEP 9: API DESIGN (IF APPLICABLE)
For each endpoint:
- Endpoint name
- Method (GET/POST/etc.)
- Request structure
- Response structure

---

### STEP 10: DATABASE DESIGN
- Tables / collections
- Fields with types
- Relationships

---

### STEP 11: SUCCESS METRICS
- How success will be measured (KPIs)

---

### STEP 12: RISKS & LIMITATIONS
- Technical risks
- Product risks
- Assumptions

---

### OUTPUT FORMAT:
- Use clean headings
- Be precise, not verbose
- Avoid generic statements
- Focus on implementation clarity

---

### INPUT:
[Paste your problem statement here]
